🔐 What is a Hash?

A hash is a one-way cryptographic function that converts any input data into a fixed-size string. It works like a “digital fingerprint.”

Example:

Input:  "password123"
MD5:    482c811da5d5b4bc6d497ffa98491e38
SHA256: ef92b778bafe771e89245b89ecbc08a44a4e166c06659911881f383d4473e94f
🎯

Deterministic

The same input always produces the same hash

🔒

One-way

It’s impossible to recover the original data from a hash

📏

Fixed size

Any size input produces a fixed-length output

💥

Avalanche effect

A tiny change creates a completely different hash

⚙️ How Hashing Works

1

Input Processing

Your text/password is taken and converted into binary data

2

Hash Algorithm Application

The selected algorithm (MD5/SHA etc.) performs mathematical operations

3

Output Generation

A fixed-length hexadecimal hash string is generated

⚠️ Important:

Hashing is a one-way process. You cannot directly recover the original password from a hash. The only way to find a match is via brute-force or dictionary attack.

🔢 Supported Hash Algorithms

MD5

⚠️ Not Secure

128-bit (32 hex characters)

A legacy algorithm — fast but cryptographically broken. Use only for checksums.

SHA-1

⚠️ Deprecated

160-bit (40 hex characters)

Better than MD5 but now deprecated. Still used internally by Git.

SHA-256

✅ Recommended

256-bit (64 hex characters)

Part of the SHA-2 family, currently the most popular and secure. Used in Bitcoin.

SHA-384

✅ Secure

384-bit (96 hex characters)

SHA-2 family — use for extra security requirements.

SHA-512

✅ Secure

512-bit (128 hex characters)

The strongest version of the SHA-2 family — for maximum security.

🛡️ Security Tips

✅ DO:

  • Only test your own passwords
  • Use strong, unique passwords
  • Use a password manager
  • Enable two-factor authentication
  • Use SHA-256 or SHA-512 for modern applications

❌ DON'T:

  • Never try to crack someone else’s hash (illegal)
  • Don’t use MD5 or SHA-1 for sensitive data
  • Don’t use common passwords
  • Don’t reuse the same password on multiple sites
  • Don’t generate sensitive hashes on public/shared computers

📚 Encoding Categories

📦 Base Encodings

Base encoding schemes are used to represent binary data in ASCII string format. They're commonly used for data transmission and storage.

Base64 Most Popular

Use case: Email attachments, data URLs, JWT tokens

Characters: A-Z, a-z, 0-9, +, /

Input:  "Hello World"
Output: SGVsbG8gV29ybGQ=
Base64 URL Safe Web Safe

Use case: URLs, filenames, cookies

Difference: Uses - and _ instead of + and /, removes padding

Base32

Use case: TOTP tokens, case-insensitive systems

Characters: A-Z, 2-7

Base58 Crypto

Use case: Bitcoin addresses, IPFS hashes

Why?: Removes confusing characters (0, O, I, l)

Base85 (Ascii85)

Use case: PDF files, PostScript

Efficiency: 25% more efficient than Base64

🔠 ASCII & Number Systems

Convert text to different number system representations for debugging, analysis, or obfuscation.

Hexadecimal (Base16)

Binary (Base2)

Use case: Low-level programming, bitwise operations

Text:   "A"

Octal (Base8)

Decimal (ASCII Codes)

🌐 URL & Network Encodings

Encodings designed for safe transmission over networks and in URLs.

URL Encoding (Percent Encoding) RFC 1738

Use case: URLs, query parameters, form data

Input:  "hello world"
Output: hello%20world
Punycode

Use case: Internationalized domain names (IDN)

Input:  "münchen.de"
Output: "xn--mnchen-3ya.de"
Quoted-Printable

Use case: Email content (MIME encoding)

Why?: Makes 8-bit data safe for 7-bit email systems

UUEncode / XXEncode

Use case: Binary files in email (legacy)

History: Unix-to-Unix Copy (UUCP) protocol

💎 Pro Tips & Best Practices

✅ Do's

  • Use Base64 for binary data in JSON/XML
  • Always URL encode query parameters
  • HTML escape all user-generated content
  • Use Base64 URL-safe for URLs and filenames
  • Test thoroughly with edge cases and special characters
  • Combine tools for complex transformations

⚠️ Don'ts

  • Don't use encoding for security — it's not encryption!
  • Don't assume Base64 is encrypted — anyone can decode it
  • Don't use classical ciphers for real security (ROT13, Caesar)
  • Don't trust decoded JWT without signature verification
  • Don't double-encode unless necessary

🔑 Key Difference: Encoding vs Encryption

Encoding
  • Purpose: Data representation
  • Reversible: Yes, easily
  • Security: None
  • Key needed: No
  • Example: Base64, URL encoding
Encryption
  • Purpose: Data protection
  • Reversible: Yes, with key
  • Security: High
  • Key needed: Yes
  • Example: AES, RSA

⚡ Performance & Limitations

💻 Browser-Based Processing

All encoding/decoding happens in your browser using JavaScript. This means:

  • ✅ Complete privacy — no data sent to servers
  • ✅ Works offline after initial page load
  • ✅ Fast for small to medium data (up to 10MB)
  • ⚠️ May slow down with very large files (100MB+)
  • ⚠️ Memory limited by browser (typically 1-2GB)

📏 Size Considerations

  • Base64 increases size by ~33% — "Hello" (5 bytes) → "SGVsbG8=" (8 bytes)
  • URL encoding varies — depends on special characters
  • Base85 is more efficient — only ~25% increase
  • Compression first — consider gzip before encoding
